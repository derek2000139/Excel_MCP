# ExcelForge 总体设计规范（V2）

**基线版本**：`v2.0.0`\
**规范版本**：`V2.4`\
**文档状态**：基线规范\
**更新日期**：`2026-03-30`

***

## 1. 文档定位

本文档以当前已完成重构的 `ExcelForge 2.0` 为唯一事实基线，定义 `V3` 阶段的总体设计约束、能力扩展方向、接口演进原则与验收口径。

`V3` 的核心目标不是再次推翻架构，而是在保留 `2.0` 多 Gateway + 共享 Runtime 体系的前提下，完成以下几类能力建设：

1. 将 `PQ` 从“占位接口”升级为“可实际使用的 Power Query 能力域”。
2. 将 `M` 语言能力作为 `PQ` 域的一部分纳入正式设计与实现范围。
3. 将当前仍未覆盖或仅支持只读查看的 Excel 高阶功能补齐，优先包括条件格式、数据验证、透视表及相关规则能力。
4. 为 `DAX / 数据模型` 能力预留正式设计边界与实施路径。

本文档作为后续所有 `V3.x` 设计文档的上位规范。除非某专题文档明确声明并论证变更点，否则默认继承本文档。

## 2. 2.0 基线快照

### 2.1 架构基线

`2.0` 已确定并落地的总体架构如下：

- `Runtime` 为独立常驻进程。
- `Gateway` 按能力域拆分为 `core / vba / recovery / pq` 四个入口。
- 进程间通信固定采用 `Named Pipe + JSON-RPC`。
- Excel COM 状态、Workbook 注册、持久化、恢复、安全控制全部只存在于 `Runtime` 内。
- `Gateway` 仅承担 MCP 工具注册、参数映射、错误转译、Runtime 自动拉起与重连。

### 2.2 当前 Gateway 与工具基线

| -------------------- | ---------------------------------------------------- | -----: |
| `excel-core-mcp`     | workbook / sheet / range / formula / format / server |     37 |
| `excel-vba-mcp`      | VBA 相关能力                                             |      6 |
| `excel-recovery-mcp` | rollback / backups / audit / names                   |      8 |
| `excel-pq-mcp`       | PQ 占位能力                                              |      5 |
| **合计**               | 共享同一 Runtime                                         | **56** |

> **V2.4 扩充**：新增 `data` bundle（table 8工具）、`analysis` bundle（6工具）、`workbook_ops` bundle（6工具）、`edit_structure` bundle（sheet 4工具+range 2工具）、`sheet_structure` bundle（5工具）、`range_structure` bundle（9工具）。

## 3. V2 目标与范围

### 3.1 总体目标

`V2` 的目标定义为：

1. **架构不再大改**：继续沿用 `2.0` 的多 Gateway 共享 Runtime 方案。
2. **PQ 与 M 语言能力从占位升级为可用**：至少完成查询读取、编辑、刷新、连接查看与 M 脚本维护。
3. **Excel 高阶规则能力补齐**：优先补齐条件格式与数据验证的写入、删除、重排、批量维护。
4. **透视表能力纳入正式范围**：至少具备检查、创建、刷新、布局调整的设计与交付路径。
5. **DAX / 数据模型能力纳入正式范围**：形成 measure 与模型对象管理的统一设计边界。
6. **恢复、安全、审计贯穿新能力**：所有新能力必须纳入统一的 backup/snapshot/audit 体系。
7. **后续专题文档可落地**：后续每一个设计文档都能直接基于本规范拆分开发与验收。

### 3.2 本阶段范围

纳入 `V2` 总体规范范围的能力：

- `Power Query`
- `M` 语言
- 条件格式
- 数据验证
- 透视表
- `DAX / 数据模型`
- 格式域增强
- 与上述能力直接相关的 Runtime API、Gateway 工具、错误码、恢复策略、测试策略

不纳入本轮强制范围，但允许在专题设计中提出的能力：

- 图表
- 表格对象（ListObject）高级管理
- Office Script / 外部脚本集成

### 3.3 非目标

`V2` 不以以下事项为目标：

- 改用 HTTP、gRPC 或其他 IPC 方案替换 Named Pipe
- 取消 Runtime 独立进程
- 脱离 Windows / Excel COM
- 为旧单体 `serve` 入口恢复兼容

## 4. 固定架构约束

以下约束在 `V2` 阶段视为固定，不应在专题设计中随意变更：

### 4.1 Runtime 唯一状态中心

- `ExcelWorker`
- `WorkbookRegistry`
- `services`
- `persistence`
- `audit`
- `snapshot / backup`

以上对象只能驻留在 `Runtime` 进程。

### 4.2 Gateway 仍为薄层

Gateway 只允许承担以下职责：

- MCP 工具注册
- 输入参数整理与轻量校验
- Runtime 方法调用
- 错误转译
- 自动连接、自动拉起、超时处理

Gateway 不允许承载：

- Excel COM 调用
- 业务决策
- 数据持久化
- Workbook 本地缓存

### 4.3 ID 统一原则

- `workbook_id` 只能由 Runtime 在 `workbook.open/create` 时分配。
- 后续 `snapshot_id / backup_id / operation_id / query_id` 等标识也必须由 Runtime 统一管理。
- 跨 Gateway 的同一工作簿必须始终使用同一 `workbook_id`。

### 4.4 通信协议固定

- 传输层：`Windows Named Pipe`
- 协议层：`JSON-RPC`
- 审计字段：`actor_id`、`client_request_id` 全链路透传
- Runtime 不感知上层 IDE，只感知标准请求上下文

## 5. V3 能力扩展原则

### 5.1 新能力必须走同一分层

所有 `V2` 新能力都必须遵循以下链路：

`Gateway Tool -> runtime_client -> Runtime Dispatcher -> runtime_api -> service -> Excel COM / persistence`

禁止出现：

- Gateway 直接访问 `services/*`
- 新专题绕开 `runtime_api`
- 新能力只在 Gateway 做局部实现

### 5.2 新能力必须声明成熟度等级

每个 `V2` 新能力都必须在设计文档中明确以下等级：

| 等级   | 含义                    |
| ---- | --------------------- |
| `L0` | 占位接口，仅返回未启用           |
| `L1` | 只读能力可用                |
| `L2` | 核心读写能力可用              |
| `L3` | 含恢复、安全、审计、错误语义的完整可用能力 |

`V2` 的总体目标是：

- `PQ` 至少达到 `L2`，推荐达到 `L3`
- `M` 语言脚本能力至少达到 `L2`
- 条件格式至少达到 `L2`
- 数据验证至少达到 `L2`
- 透视表至少达到 `L1`，推荐达到 `L2`
- `DAX / 数据模型` 至少达到 `L1`，推荐达到 `L2`

### 5.3 新能力优先兼容现有工具名

- 已上线工具名默认稳定。
- 能通过扩展现有 `action` 完成的，优先扩展现有工具。
- 仅当现有工具的 Schema 会明显失控时，才新增独立工具。
- 一旦新增独立工具，必须在设计文档中给出“不复用现有工具”的理由。

## 6. PQ 与 M 语言能力域设计要求

### 6.1 V2 对 PQ 的最低目标

`excel-pq-mcp` 在 `V2` 中不得继续停留在纯占位状态，至少应完成以下真实能力：

| 工具                    | 目标状态 | 说明                               |
| --------------------- | ---- | -------------------------------- |
| `pq.list_queries`     | 可用   | 列出工作簿中的 Query 名称、加载方式、刷新状态       |
| `pq.get_code`         | 可用   | 读取指定 Query 的 M Code              |
| `pq.update_query`     | 可用   | 更新已有 Query 的 M Code              |
| `pq.refresh`          | 可用   | 刷新单个 Query 或支持刷新全部               |
| `pq.list_connections` | 可用   | 列出 Workbook Connection 及关联 Query |

### 6.2 PQ 的实现边界

`V2` 第一阶段 PQ 设计必须遵循以下边界：

- 以“操作现有工作簿内的 Query / Connection”为主。
- 不强制要求在第一阶段支持“新建外部数据源连接向导”。
- 不在第一阶段承担凭据管理 UI 自动化。
- 不通过 Gateway 直接操作外部网络；外部数据访问仅允许经 Excel / Power Query 引擎触发。

### 6.3 M 语言能力要求

`M` 能力在 `V2` 中视为 `PQ` 域的一部分，后续专题设计必须满足以下要求：

- `pq.get_code` 返回完整 M 脚本，而不是裁剪后的片段。
- `pq.update_query` 必须支持完整脚本替换，不能只支持模板参数替换。
- 必须允许使用标准 `M` 函数、`let ... in ...` 结构、嵌套函数和多步骤表达式。
- 如首版不支持 `M` 脚本校验或依赖分析，必须在文档中明确列为后续阶段能力。

推荐后续扩展能力：

- `pq.validate_m_code`
- `pq.inspect_dependencies`
- `pq.format_m_code`

### 6.4 PQ 的 Runtime 设计要求

后续专题设计中必须补齐以下 Runtime 组件：

- `PqService`
- `runtime_api/pq_api.py` 的真实实现
- Query / Connection 元数据模型
- PQ 刷新超时控制
- PQ 操作的审计和恢复策略

### 6.5 PQ 恢复策略要求

| 操作                    | 恢复策略                      |
| --------------------- | ------------------------- |
| `pq.list_queries`     | `none`                    |
| `pq.get_code`         | `none`                    |
| `pq.list_connections` | `none`                    |
| `pq.update_query`     | `backup`                  |
| `pq.refresh`          | 默认 `backup`，如专题文档另有优化必须论证 |

原因：

- Query 代码变更可能影响加载结果、连接定义、下游表结构。
- 刷新可能直接改写工作表表格、数据模型、缓存结果。

### 6.6 PQ 错误语义要求

`V2` 的 PQ 设计文档必须定义至少以下错误类别：

- Query 不存在
- Connection 不存在
- Power Query 不可用或版本不支持
- 刷新超时
- 刷新失败
- Query 代码非法
- Workbook 当前格式或状态不支持 PQ 变更

## 7. 条件格式与数据验证设计要求

### 7.1 当前缺口

当前 `2.0` 已支持：

- `sheet.get_conditional_formats`
- `sheet.get_data_validations`

但仍缺少：

- 条件格式创建
- 条件格式更新
- 条件格式删除
- 条件格式优先级调整
- 数据验证创建
- 数据验证更新
- 数据验证删除

### 7.2 V3 目标

`V2` 需要将“规则查看”升级为“规则管理”，至少达到以下能力：

| 能力    | 最低要求               |
| ----- | ------------------ |
| 条件格式  | 支持新增、覆盖、删除、读取、重排   |
| 数据验证  | 支持新增、覆盖、删除、读取      |
| 批量范围  | 支持按范围批量应用          |
| 规则可追溯 | 返回规则标识、适用区域、类型、优先级 |
| 恢复能力  | 写操作纳入 backup       |

### 7.3 建议的工具演进方向

在不破坏当前只读工具的前提下，`V2` 建议采用以下演进方向之一：

#### 方案 A：扩展 `format.manage`

继续使用动作式工具，新增 `action`：

- `set_conditional_formats`
- `clear_conditional_formats`
- `set_data_validation`
- `clear_data_validation`
- `reorder_conditional_formats`

优点：

- 工具数增长最小
- 与当前 `format.manage` 风格一致

缺点：

- Schema 会变复杂

#### 方案 B：新增规则工具

新增独立工具：

- `format.manage_conditional_formats`
- `format.manage_data_validations`

优点：

- 语义清晰
- 参数模型更容易收敛

缺点：

- 工具数量增加

`V2` 总体规范不强制二选一，但后续专题设计必须明确采用哪一方案，并给出理由。

### 7.4 条件格式分阶段支持要求

由于 Excel 条件格式类型较多，专题设计允许按阶段实现，但必须明确阶段边界：

**第一优先级**

- 基于单元格值的比较规则
- 基于公式的规则
- 文本包含/日期发生/空值等常用规则

**第二优先级**

- 色阶
- 数据条
- 图标集

若首版不支持第二优先级类型，必须：

- 在接口中显式返回“不支持的规则类型”
- 在文档中列出兼容矩阵
- 避免 silent failure

### 7.5 数据验证分阶段支持要求

首版应至少支持：

- 整数/小数范围
- 序列（List）
- 日期
- 文本长度
- 自定义公式

同时支持以下附加属性：

- `allow_blank`
- `show_input_message`
- `prompt_title`
- `prompt`

## 8. 格式域与高级 Excel 功能补齐要求

### 8.1 格式域原则

`format` 域在 `V2` 中不再只视为“简单样式设置”，而是负责所有“与单元格呈现规则直接相关”的能力集合。

至少应覆盖：

- 基础样式
- 自动列宽
- 条件格式
- 数据验证

### 8.2 与 Sheet 域的职责边界

- `sheet.*` 负责工作表结构和只读检查视图。
- `format.*` 负责规则类写操作与样式类写操作。
- 只读查看接口可以保留在 `sheet.*`，但写接口应尽量集中在 `format.*`。

### 8.3 与 PQ 的联动要求

因为 PQ 常见加载结果会落地到工作表表格区域，后续设计需要考虑以下联动：

- Query 刷新后条件格式是否保留
- Query 刷新后数据验证是否保留
- Query 刷新导致行列扩展时规则覆盖范围如何处理
- Query 刷新失败时是否触发恢复或回滚提示

***

## 9. 透视表与数据模型 / DAX 设计要求

### 9.1 透视表能力目标

`V2` 需要将透视表能力纳入正式设计范围，最低目标包括：

- 列出工作簿中的透视表对象
- 查看透视表数据源、字段布局、过滤器与目标区域
- 创建或重建透视表
- 刷新透视表或透视缓存
- 调整行、列、值、筛选字段布局

如首版无法覆盖全部写能力，至少应先实现“检查 + 刷新 + 基础创建”。

### 9.2 透视表恢复策略要求

| 操作         | 恢复策略     |
| ---------- | -------- |
| 列出 / 检查    | `none`   |
| 刷新透视表      | `backup` |
| 创建 / 重建透视表 | `backup` |
| 调整布局       | `backup` |

### 9.3 DAX / 数据模型能力目标

`V2` 需要将 `DAX / 数据模型` 作为独立能力域纳入专题设计。首版至少应覆盖以下方向：

- 列出模型表、列、Measure
- 读取现有 Measure 的 DAX 表达式
- 创建或更新 Measure
- 删除 Measure
- 触发模型刷新或与工作簿刷新流程协同

如首版不支持计算列、层级、KPI 或关系管理，必须在专题文档中明确列为后续阶段。

### 9.4 DAX / 数据模型的命名建议

总体规范建议后续专题优先使用如下域命名之一：

- Runtime API：`model.*`
- 对外工具：`model.*` 或 `dax.*`

最终采用哪一套命名，由专题文档决定；但必须保证同一能力域内命名风格一致。

### 9.5 与 PQ、透视表的联动要求

后续专题设计必须考虑以下联动：

- Query 刷新后透视缓存是否需要同步刷新
- Query 结构变化后 Measure / 透视字段是否失效
- 数据模型刷新失败时的错误传播
- DAX 表达式引用对象被 Query 变更影响时的校验与报错

## 10. 接口演进与兼容性要求

### 10.1 公共输入输出规范

后续专题设计必须继续沿用 `2.0` 统一响应信封：

- `success`
- `code`
- `message`
- `data`
- `warnings`
- `meta`
- `recovery`

### 10.2 新增方法的命名要求

- Runtime 方法名：`domain.method`
- Gateway 工具名：优先延续现有命名风格
- 动作式工具优先使用 `action`
- 不允许重新引入旧单体时代的混乱别名

### 10.3 破坏性变更门槛

如果后续专题设计要修改已有工具名、参数名、响应结构，必须同时提供：

1. 变更原因
2. 兼容影响面
3. 迁移方式
4. 回退方式

无上述四项，不允许定义破坏性变更。

## 11. 恢复、安全、审计统一要求

### 11.1 恢复要求

所有 `V2` 新增写操作都必须明确恢复策略：

| 类型                | 推荐策略        |
| ----------------- | ----------- |
| 纯读取               | `none`      |
| 局部单元格规则写入         | `backup` 优先 |
| Query / M 脚本代码与刷新 | `backup`    |
| 透视表与数据模型写操作       | `backup`    |
| 结构性修改             | `backup`    |

如要使用 `snapshot` 替代 `backup`，必须证明：

- 影响范围局限且可精确恢复
- 不会因结构变化导致快照失效

### 11.2 安全要求

针对 `PQ / M`、透视表、`DAX / 数据模型` 与高级规则能力，专题设计必须额外说明：

- 外部连接/刷新是否允许
- 是否会触发凭据、隐私级别或安全提示
- 如何限制高风险外部数据源
- 刷新超时与失败后的处理

### 11.3 审计要求

所有新操作必须记录：

- `actor_id`
- `client_request_id`
- `workbook_id`
- `tool_name`
- `success`
- `recovery_strategy`
- `recovery_reference_id`

## 12. 测试与验收要求

### 12.1 必测维度

所有 `V2` 新专题至少覆盖：

- 契约测试
- Runtime API 测试
- Gateway 映射测试
- Excel 集成测试
- 恢复链路测试
- 超时与异常测试

### 12.2 PQ 必测场景

- 打开含 Query 的工作簿后列出 Query
- 读取指定 Query 的代码
- 更新 Query 代码并保存工作簿
- 刷新指定 Query
- 刷新失败后的错误码与审计记录
- Query 更新/刷新后的备份恢复

### 12.3 条件格式/数据验证必测场景

- 创建规则
- 读取规则
- 更新规则
- 删除规则
- 多规则优先级变更
- 规则应用区域扩展/收缩
- 恢复后规则状态一致

### 12.4 透视表与 DAX 必测场景

- 列出透视表与数据源关系
- 创建或刷新透视表
- 调整透视表布局后校验结果
- 列出模型 Measure
- 创建或更新 Measure 并再次读取
- Query 刷新后透视表与 Measure 的联动校验
- 对相关写操作执行 backup 恢复并验证状态一致

### 12.5 验收定义

只有同时满足以下条件，专题能力才可视为完成：

1. 工具在对应 Gateway 中可实际调用。
2. Runtime API 有真实实现，不再返回占位错误。
3. 审计链路完整。
4. 写操作可恢复。
5. 集成测试可重复通过。
