# V2.2.1 Runtime 稳定性与进程治理修复计划（修订版）

***

## 1. 修复目标

根据设计文档 V2.2.1，本次修复 5 个 Runtime 层运行时问题。不改变 V2.2 的架构决策（单入口、Profile/Bundle 装配、64 个工具注册）。

| 编号   | 问题                       | 严重度  | 根因                                               | 状态    |
| ---- | ------------------------ | ---- | ------------------------------------------------ | ----- |
| T0   | Gateway 重启不带动 Runtime 重启 | 🔴 高 | Runtime 独立进程，Gateway 重启后仍复用旧 Runtime             | ✅ 已完成 |
| P0.1 | 打开文件创建第二个 Excel 实例       | 🔴 高 | `ensure_app()` 复用逻辑有问题，且 `_is_app_valid()` 检测不完善 | ✅ 已完成 |
| P1   | Excel 僵尸进程堆积             | 🔴 高 | rebuild 只创建新进程，不终止旧进程，无 PID 追踪，无重建锁              | ✅ 已完成 |
| P2   | 启动或非文件操作时弹出空白窗口          | 🟡 中 | Excel 初始化时 Visible 过早设为 True，且未关闭自动创建的 Book1     | ✅ 已完成 |
| P3   | 手动关闭窗口后重新打开会恢复所有旧文件      | 🟡 中 | Registry 状态未随 Excel 进程消亡同步清理，rebuild 时盲目恢复       | ✅ 已完成 |
| P4   | 无法查看操作日志                 | 🟡 中 | 日志仅输出到 MCP stdio 的 stderr，无文件持久化                 | ✅ 已完成 |
| P5   | 无效 COM 句柄未清理             | 🟡 中 | Registry 不检查 COM 句柄健康状态，失效句柄直到被访问才报错             | ✅ 已完成 |

## 2. 实施步骤

### ⚠️ T0（最高优先级）：Gateway 重启不带动 Runtime 重启

**问题**：Gateway 重启时会"复用"已有 Runtime，导致新代码不生效。

**解决方案**：在 `host.py` 增加 `--restart-runtime` 参数，支持 `always` / `if-stale` / `never` 三种策略。

**状态**：✅ 已完成

| 任务   | 操作 | 文件                          | 说明                                                         |
| ---- | -- | --------------------------- | ---------------------------------------------------------- |
| T0-1 | 修改 | `host.py`                   | 增加 `--restart-runtime` 参数和 `ensure_runtime_fresh()` 逻辑     |
| T0-2 | 修改 | `runtime_client_manager.py` | 补充 `get_runtime_pid()`, `shutdown()`, `start_runtime()` 方法 |

**开发期建议**：在 `mcp.json` 的 args 中加上 `--restart-runtime always`

**参考文档**：`new.md` - Gateway 重启不带动 Runtime 重启的问题

## 4. 验收标准

### 4.1 P1 僵尸进程

| 场景        | 操作                      | 期望               |
| --------- | ----------------------- | ---------------- |
| 正常使用      | MCP 打开文件 → 操作 → 关闭      | 0 或 1 个 Excel 进程 |
| 手动关窗口后重开  | MCP 打开 → 手动关窗口 → MCP 打开 | 仅 1 个进程          |
| 连续关窗口 5 次 | 循环：MCP 打开 → 手动关 × 5     | 始终 ≤1 个进程        |
| 启动时清理     | 人为留 2 个孤儿进程 → 启动 MCP    | 孤儿被清理，日志中有记录     |

### 4.2 P2 空白窗口

| 场景      | 操作                         | 期望               |
| ------- | -------------------------- | ---------------- |
| 首次启动    | 启动 MCP，不执行任何命令             | 无 Excel 窗口弹出     |
| 非文件操作   | 执行 `get_status` 或 `health` | 无窗口弹出            |
| 打开文件    | 执行 `workbook.open_file`    | Excel 窗口出现，显示该文件 |
| 打开第二个文件 | 再次 `open_file` 不同文件        | 同一窗口打开第二个文件      |

### 4.3 P3 不恢复旧文件

| 场景            | 操作                         | 期望       |
| ------------- | -------------------------- | -------- |
| 手动关窗口后重开      | 打开 A 和 B → 手动关窗口 → 打开 C    | 只有 C 被打开 |
| Registry 状态一致 | 上述操作后 `workbook.list_open` | 只列出 C    |

### 4.4 P4 日志文件

| 场景          | 操作           | 期望                                    |
| ----------- | ------------ | ------------------------------------- |
| 日志文件生成      | 启动 MCP       | `~/.excelforge/logs/` 下存在当天日志文件       |
| 内容完整        | 执行几个工具调用     | 日志含 TOOL CALL / TOOL OK 记录            |
| rebuild 可追踪 | 触发一次 rebuild | 日志含 REBUILD START / COMPLETE 及 PID 信息 |

### 4.5 P5 COM 句柄清理

| 场景       | 操作                      | 期望                                 |
| -------- | ----------------------- | ---------------------------------- |
| 失效句柄友好提示 | 打开 A → 手动关窗口 → 对 A 执行操作 | 返回"请重新打开"而非裸 COMError              |
| 自动清理     | 打开 A → 手动关窗口 → 等待 60 秒  | 日志中出现 stale handle 清理记录            |
| 计数一致     | 正常打开 2 个文件 → 检查统计       | registry\_count = excel\_count = 2 |
