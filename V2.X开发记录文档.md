## 核心根因：Windows 上 os.kill(pid, 0) 的 C 层异常状态污染

### 问题元信息

- **首次出现版本**: v2.0 及更早版本（所有版本）
- **回退到v2.0后是否仍复现**: 是
- **是否需全重启缓解**: 是（但根因在 lifecycle.py）
- **怀疑层级**: Runtime client / lifecycle / Windows API
- **状态**: ✅ 已修复

### 问题描述

执行 `python -m excelforge.runtime` 启动 Runtime 进程后，进程立即退出，没有任何错误输出。但 lock 文件存在，说明 `write_runtime_lock()` 执行成功，`serve_forever()` 应该阻塞。

调用任何 MCP 工具时返回错误：
```
E500_INTERNAL: <built-in function kill> returned a result with the exception set
E503_RUNTIME_UNAVAILABLE: Runtime lock file not found or process is not alive
```

### 问题链路

```
Gateway 调用 call()
  → _ensure_connected()
    → _wait_for_runtime_lock()
      → read_runtime_lock_from_dir()
        → is_process_alive(pid)
          → os.kill(pid, 0)    ← Windows 上行为异常
            → 不是抛 OSError，而是设置了 C 层异常状态
            → 返回到 Python 层时异常状态未清除
            → 后续任何 C 扩展调用都会报
               "<built-in function kill> returned a result
                with the exception set"
```

### 根本原因

`os.kill(pid, 0)` 在 Windows 上是一个已知的坑：
- Windows 没有 POSIX 信号机制
- Python 的 `os.kill` 在 Windows 上调用 `TerminateProcess` 或 `OpenProcess`
- 信号 0 的处理不一致
- 某些 Python 版本 + Windows 组合下会产生 C 层异常状态污染
- 这个异常状态被后续所有 C 扩展调用继承

### 修复方案

修改 `excelforge/runtime/lifecycle.py`，将 `is_process_alive()` 改为 Windows 兼容实现：

```python
def is_process_alive(pid: int) -> bool:
    """检查指定 PID 的进程是否存在。Windows 兼容实现。"""
    if pid <= 0:
        return False

    if sys.platform == "win32":
        return _is_process_alive_windows(pid)
    else:
        return _is_process_alive_posix(pid)


def _is_process_alive_windows(pid: int) -> bool:
    """Windows 实现：使用 ctypes 调用 OpenProcess + GetExitCodeProcess。"""
    import ctypes
    import ctypes.wintypes

    PROCESS_QUERY_LIMITED_INFORMATION = 0x1000
    STILL_ACTIVE = 259

    kernel32 = ctypes.windll.kernel32

    handle = kernel32.OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION, False, pid)
    if handle == 0:
        return False

    try:
        exit_code = ctypes.wintypes.DWORD()
        if kernel32.GetExitCodeProcess(handle, ctypes.byref(exit_code)):
            return exit_code.value == STILL_ACTIVE
        else:
            return True
    finally:
        kernel32.CloseHandle(handle)


def _is_process_alive_posix(pid: int) -> bool:
    """POSIX 实现：使用 os.kill(pid, 0)。"""
    try:
        os.kill(pid, 0)
        return True
    except ProcessLookupError:
        return False
    except PermissionError:
        return True
    except OSError:
        return False
```

### 修复文件

- `excelforge/runtime/lifecycle.py` - 替换 `is_process_alive()` 函数

### 修复验证

| 工具 | 修复前 | 修复后 |
|------|--------|--------|
| `server.get_status` | ❌ E500_INTERNAL | ✅ 正常 |
| `workbook.open_file` | ❌ E500_INTERNAL | ✅ 正常 |
| `workbook.create_file` | ❌ E500_INTERNAL | ✅ 正常 |
| `vba.inspect_project` | ❌ E500_INTERNAL | ✅ 正常 |

### 影响范围

- Runtime 启动即退出 → 实际上是 Gateway 无法连接 Runtime
- VBA 全部 OSError → 实际上是 Gateway 无法连接 Runtime
- Recovery/Rollback OSError → 实际上是 Gateway 无法连接 Runtime

## 内存溢出问题

### 问题元信息

- **首次出现版本**: v2.0测试期
- **回退到v2.0后是否仍复现**: 是
- **是否需全重启缓解**: 是
- **怀疑层级**: Worker / COM
- **状态**: 待排查

### 问题描述

在使用 MCP Excel 工具对 TEST.xlsm 进行操作时，出现内存溢出错误。

具体错误发生在尝试执行 `vba.sync_module` 创建新的 VBA 模块 `modBeautifySales` 时，系统返回错误：

```
E500_INTERNAL: <built-in function kill> returned a result with the exception set
```

后续再次尝试时出现：

```
E503_RUNTIME_UNAVAILABLE: Runtime lock file not found or process is not alive
```

### 涉及的操作

1. 打开 TEST.xlsm 工作簿
2. 创建 VBA 模块 `modBeautify`（147行代码）
3. 创建 VBA 模块 `Test`（175行代码，包含 SplitByClass 等多个过程）
4. 执行 SplitByClass 宏，按班级拆分为15个工作表
5. 每个班级工作表都应用了美化格式（背景色、边框、斑马纹等）
6. 创建新工作簿 NewFile.xlsx
7. 尝试为 NewFile 创建 VBA 模块时失败

### 问题原因

1. **工作簿数据量大**：TEST.xlsm 包含16个工作表，每个表都有大量格式化样式

2. **VBA模块多次操作**：连续创建多个VBA模块，COM对象可能未及时释放

3. **隐藏Excel进程**：运行模式 `desktop_agent_hidden_excel` 下，隐藏的Excel实例处理大量VBA操作时更容易出现内存问题

4. **循环引用风险**：VBA代码中的Range对象等COM对象可能存在未及时释放的情况

### 备注

**注意**：早期报告的 `<built-in function kill>` 错误实际上是由 Windows `os.kill(pid, 0)` 问题导致的，修复 `lifecycle.py` 后已解决。但真正的内存溢出问题可能仍然存在，需要单独排查。

## Excel Worker 首次启动超时问题

### 问题元信息

- **首次出现版本**: v2.0测试期
- **回退到v2.0后是否仍复现**: 是
- **是否需全重启缓解**: 否（等待首次启动完成即可）
- **怀疑层级**: Worker / bootstrap
- **状态**: 待排查

### 问题描述

使用 MCP Excel 工具执行首次操作（如 `workbook.create_file`）时，返回错误：

```
E503_RUNTIME_UNAVAILABLE: Runtime lock file not found or process is not alive
```

部分情况下耗时超过 40 秒才返回超时错误。

### 问题原因

1. **懒加载模式**：Excel Worker 在 `bootstrap.create_runtime_services()` 时仅创建实例，不启动 Excel 进程
2. **首次请求才启动**：当第一个请求到达时，`excel_worker.py:109-110` 才调用 `self.start()` 启动工作线程
3. **COM 对象创建慢**：首次调用 `ensure_app()` 创建 Excel COM 实例需要较长时间（~40 秒）
4. **超时限制**：默认 `operation_timeout_seconds: 30`，首次请求等待 COM 初始化的时间超过此限制

### 影响范围

所有首次请求都会受影响，包括：
- `workbook.create_file`
- `workbook.open_file`
- `sheet.create` 等需要 Excel 的操作

### 触发条件

服务刚启动后，第一个发往 Excel Worker 的请求。

## VBA 工具执行 Bug

### 问题元信息

- **首次出现版本**: v2.0测试期
- **回退到v2.0后是否仍复现**: 是（表象已修复）
- **是否需全重启缓解**: 是（表象）
- **怀疑层级**: Runtime 多进程隔离
- **状态**: ✅ 根因已确认（多进程隔离问题）

### 问题描述

V2.0 时代所有 VBA 相关工具返回内部错误：

```
E500_INTERNAL: <class 'OSError'> returned a result with an exception set
```

V2.1 修复了 `os.kill(pid, 0)` 问题后：
- ✅ `vba.inspect_project` - 正常工作
- ✅ `vba.sync_module` - 正常工作
- ✅ `vba.remove_module` - 正常工作
- ❌ `vba.execute` - 仍失败

### 实际验证记录

**测试场景**：为 TEST.xlsm（考试成绩表）创建图表生成模块

**操作序列**：
1. `workbook.open_file` 打开 TEST.xlsm - ✅ 成功
2. `sheet.inspect` 检查数据结构 - ✅ 成功
3. `range.read_values` 读取数据 - ✅ 成功
4. `vba.sync_module` 创建 modChartGenerator（135行代码）- ✅ 成功
5. `vba.execute` 执行 创建成绩图表 宏 - ❌ 失败

**错误信息**：
```
E410_WORKBOOK_STALE: Workbook handle is stale after Excel rebuild; reopen with workbook.open_file
```

### 根因分析

**核心问题**：`excel-core-mcp` 和 `excel-vba-mcp` 是两个独立的 Runtime 进程，各自拥有独立的 Excel 实例。

```
excel-core-mcp → Runtime A → Excel 实例 A (打开 TEST.xlsm)
excel-vba-mcp  → Runtime B → Excel 实例 B (空白)
                              ↑
                        health_ping 失败
                        → Workbooks.Count = 0
                        → 触发 rebuild
                        → E410_WORKBOOK_STALE
```

### 直接验证

1. **系统进程检查**：执行 `vba.execute` 时，发现有 2 个 Excel 进程
   - PID 31168: Core MCP 的 Excel（打开 TEST.xlsm）
   - PID 16552: VBA MCP 的 Excel（空白）

2. **独立 Python 脚本测试**：使用单个 `DispatchEx` 实例模拟 MCP 流程
   - ✅ 成功执行 VBA 宏
   - 证明 COM 调用本身没有问题

3. **结论**：问题不在 COM 或 VBA 本身，而在多进程隔离

### 错误影响

- ❌ 无法通过 MCP 执行任何 VBA 宏代码
- ✅ `vba.execute` 和 `vba.execute_inline` 均失败
- ✅ 其他 VBA 操作（inspect、sync_module、remove_module）正常工作
- ⚠️ VBA 代码已写入工作簿，但无法直接运行

### 临时解决方案

1. VBA 模块已成功写入工作簿，用户可手动在 Excel 中运行宏
2. 按 `Alt+F8` 打开宏对话框，选择并执行宏
3. 或在 Excel VBA 编辑器中直接运行

### 修复方向

参见" MCP 多进程隔离问题"章节，推荐方案是将所有 MCP 服务合并为单一 Runtime 进程。

### 问题元信息

- **首次出现版本**: v2.0测试期
- **回退到v2.0后是否仍复现**: N/A（已修复）
- **是否需全重启缓解**: N/A

- **状态**: ✅ 已修复

### 问题描述

### 问题原因

`sheet_service.py` 中 `create_sheet` 方法在 `position="last"` 时：

```python
ws = wb.Sheets.Add(After:=wb.Sheets(wb.Sheets.Count))  # 总是添加到最后
```

```python
active_sheet = wb.ActiveSheet
ws = wb.Sheets.Add(After:=active_sheet)  # 添加到当前激活页后面
```

- `excelforge/services/sheet_service.py` 第 205-206 行

#### 问题元信息

- **首次出现版本**: v2.1开发期
- **是否需全重启缓解**: 否（但会短暂失败）
- **怀疑层级**: Runtime / bootstrap

```
workbook.open_file 调用
  → Runtime API 处理
    → Excel Worker 处理
      → 首次 ensure_app() 耗时过长
```

3. **超时配置不合理**：默认 30 秒对于首次创建 COM 对象可能不够

#### 临时解决方案

重试一次通常会成功（因为 Excel 进程已经启动）

### Excel 进程残留（僵尸进程）

#### 问题元信息

- **首次出现版本**: v2.1开发期（实际从 v2.0 就存在）
- **是否需全重启缓解**: 是
- **怀疑层级**: Runtime / Excel Worker
- **状态**: 🔍 部分验证（与操作序列相关）

#### 问题描述

通过 MCP 打开并关闭工作簿后，Excel 进程没有退出，变成"僵尸进程"：

- 进程存在，窗口关闭
- CPU 使用率为 0 或极低
- 无法通过正常方式交互

#### 观察到的现象

**现象1：操作后产生多个残留进程**

执行以下操作序列后，产生多个残留进程：
1. MCP 打开 TestVBA.xlsm
2. 手动关闭窗口
3. MCP 打开 TEST.xlsm
4. 执行 vba.execute（失败）
5. 执行 vba.sync_module（创建 modChartGenerator）

结果：系统中出现 2-3 个残留 Excel 进程（PID 19588, 20296, 27356）

**现象2：干净环境下无残留**

执行以下操作序列后，只有 1 个正常进程：
1. 清理所有 Excel 进程
2. MCP 打开 TEST.xlsm
3. 执行 vba.sync_module（创建 modChartGenerator）
4. 执行 vba.execute（失败）

结果：系统中只有 1 个 Excel 进程（PID 12052），无残留

**结论**：进程残留问题可能与以下因素相关：
- 操作序列的复杂性
- 失败的操作（vba.execute 失败）可能导致状态异常
- 多个工作簿打开/关闭的组合

#### 可能原因

1. **COM 对象未正确释放**：Python 的 win32com 客户端没有调用 `Quit()`
2. **Excel 被设置为隐藏模式**：`Application.Visible = False` 导致窗口关闭但进程保持
3. **Python 垃圾回收延迟**：COM 对象被 Python 引用持有，无法真正退出

#### 风险

- 长期积累导致系统资源浪费
- 多个残留进程给用户造成困扰
- 可能影响后续 MCP 操作的稳定性

### MCP 与手动 Excel 共存冲突

#### 问题元信息

- **首次出现版本**: v2.1开发期
- **是否需全重启缓解**: 是
- **怀疑层级**: Runtime / Excel Worker
- **状态**: 🔍 待排查

#### 问题描述

当系统中同时存在：
1. MCP 启动的 Excel 进程
2. 用户手动打开的 Excel 窗口

MCP 的操作可能出现不可预期的行为。

#### 观察到的现象

- MCP `workbook.list` 只显示 MCP 打开的工作簿
- 手动打开的 Excel 不受 MCP 管理
- 两种 Excel 进程可能共享同一个 COM 实例，也可能各自独立

#### 潜在问题

1. **COM 对象竞争**：多个 Excel 实例竞争系统 COM 资源
2. **身份混淆**：MCP 难以区分哪个 Excel 进程属于自己
3. **窗口焦点混乱**：手动操作可能打断 MCP 的自动化流程

#### 建议

- MCP 运行时，用户应避免手动打开 Excel
- 或要求 MCP 提供"独占模式"，在启动时检测并关闭所有现有 Excel 进程

## MCP 多进程隔离问题（核心根因）

### 问题元信息

- **首次出现版本**: v2.1开发期
- **是否需全重启缓解**: 是
- **怀疑层级**: Runtime 架构
- **状态**: ✅ 根因已确认

### 问题描述

`excel-core-mcp` 和 `excel-vba-mcp` 是**两个独立的 Runtime 进程**，各自拥有独立的 Excel 实例。当通过 Core MCP 打开工作簿后，VBA MCP 无法访问该工作簿，导致 VBA 执行失败。

### 架构分析

```
┌─────────────────────────────────────────────────────────────┐
│                     用户环境                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────┐    ┌─────────────────────┐       │
│  │  excel-core-mcp     │    │  excel-vba-mcp      │       │
│  │  (Runtime 进程 A)   │    │  (Runtime 进程 B)   │       │
│  │                     │    │                     │       │
│  │  Excel 实例 A        │    │  Excel 实例 B       │       │
│  │  (打开 TEST.xlsm)   │    │  (空白实例)         │       │
│  │                     │    │                     │       │
│  │  PID: 31168         │    │  PID: 16552         │       │
│  │  Workbooks.Count=1  │    │  Workbooks.Count=0 │       │
│  └─────────────────────┘    └─────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 问题链路

1. 用户调用 `workbook.open_file` → 由 **excel-core-mcp** 处理
   - Core Runtime 创建 Excel 实例 A
   - 在实例 A 中打开 TEST.xlsm
   - 工作簿句柄与 Runtime A 关联

2. 用户调用 `vba.execute` → 由 **excel-vba-mcp** 处理
   - VBA Runtime 有自己的 Excel 实例 B（空白实例）
   - 实例 B 中**没有打开任何工作簿**

3. VBA 执行流程
   ```
   vba.execute
     → VBA Runtime B.health_ping()
       → Workbooks.Count = 0  ← 实例 B 是空白的！
       → ping 失败 → 触发 rebuild
       → 创建新 Excel 实例
     → 最终返回 E410_WORKBOOK_STALE
   ```

### 验证方法

1. 通过 PowerShell 检查系统中的 Excel 进程：
   ```powershell
   Get-Process -Name "Excel" | Select-Object Id, ProcessName, MainWindowTitle
   ```
   发现确实有**多个 Excel 进程**

2. 直接 Python 脚本模拟 MCP 流程成功：
   - 使用单个 `DispatchEx` 实例
   - 在同一实例中打开工作簿并执行 VBA
   - ✅ 成功

3. 但在 MCP 环境中失败：
   - Core MCP 和 VBA MCP 使用不同的 Runtime 进程
   - 各自拥有独立的 Excel COM 实例
   - 工作簿无法跨进程共享

### 涉及文件

| 文件 | 说明 |
|------|------|
| `excel-core-mcp.yaml` | Core MCP 配置 |
| `excel-vba-mcp.yaml` | VBA MCP 配置 |
| `runtime-config.yaml` | 共享的 Runtime 配置 |
| `excelforge/runtime/excel_worker.py` | Excel Worker 管理 |
| `excelforge/runtime/excel_app.py` | Excel COM 实例管理 |

### 修复方向

#### 方案 A：进程合并（推荐）

让所有 MCP 服务使用**同一个 Runtime 进程**，从而共享同一个 Excel 实例：

```
┌─────────────────────────────────────────────────────────────┐
│                     用户环境                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              excelforge-runtime                    │   │
│  │              (单一 Runtime 进程)                   │   │
│  │                                                     │   │
│  │  Excel 实例                                          │   │
│  │  (所有 MCP 服务共享)                                 │   │
│  │                                                     │   │
│  │  - workbook operations (via Core)                    │   │
│  │  - VBA execution (via VBA)                          │   │
│  │  - Recovery/Backup (via Recovery)                   │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 方案 B：工作簿跨进程传递

如果必须保持多进程架构，需要在工作簿打开时将其信息传递给 VBA MCP：
- 通过 COM 命名管道或其他 IPC 机制
- VBA MCP 连接到 Core MCP 的 Excel 实例

#### 方案 C：VBA MCP 代理 Core MCP 的工作簿

VBA MCP 不再拥有自己的 Excel 实例，而是通过 Core MCP 执行所有 Excel 操作：
- 所有 VBA 操作通过 Core MCP 的 Runtime 执行
- VBA MCP 作为纯逻辑层，工作簿操作委托给 Core

### 影响范围

此问题是 **vba.execute 失败的直接原因**，也间接导致：
- E410_WORKBOOK_STALE 频繁触发
- Excel 进程残留（多个 Runtime 各创建实例）
- 用户体验混乱（多个 Excel 窗口）

### 开发过程中发现的问题

#### 1. Python 闭包变量捕获问题

**问题描述**：在循环中创建闭包函数时，所有闭包共享同一个变量引用，导致所有 handler 使用最后一次迭代的值。

**错误表现**：
```
TypeError: 'server.get_status' is not a callable object
```

**修复方案**：使用工厂函数模式，每次迭代创建独立的闭包作用域：

```python
def make_tool_handler(runtime_client, tool, method):
    def handler(**kwargs):
        return call_runtime(runtime_client, tool_name=tool, method=method, params=kwargs)
    return handler

for tool_name in enabled_tools:
    runtime_method = TOOL_MANIFEST_MAP.get(tool_name, tool_name)
    handler = make_tool_handler(runtime, tool_name, runtime_method)
    mcp.tool(name=tool_name)(handler)
```

**修复文件**：`excelforge/gateway/host.py`

#### 2. FastMCP add_tool 参数顺序问题

**问题描述**：`mcp.add_tool(name, description, fn)` 的参数顺序与预期不符，第二个参数是 description 而非 fn。

**修复方案**：使用装饰器模式 `mcp.tool(name=tool_name)(handler)` 替代 `mcp.add_tool()`。

**修复文件**：`excelforge/gateway/host.py`

#### 3. Profile tool_budget 配置不合理

**问题描述**：部分 Profile 的 tool_budget 设置过低，导致启动时产生警告。

**修复方案**：调整各 Profile 的 tool_budget：

| Profile | 原预算 | 新预算 | 实际工具数 |
|---------|--------|--------|-----------:|
| basic_edit | 15 | 30 | 25 |
| calc_format | 18 | 40 | 40 |
| automation | 12 | 20 | 18 |
| legacy_core | 25 | 40 | 40 |
| legacy_vba | 12 | 20 | 18 |
| legacy_recovery | 8 | 15 | 11 |
| legacy_pq | 10 | 18 | 11 |

**修复文件**：`excelforge/gateway/profiles.yaml`
